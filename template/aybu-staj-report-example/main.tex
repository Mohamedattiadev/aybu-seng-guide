\section{GENERAL REPORT}

\subsection{Introduction}

\subsubsection{Information about the Institution}
ACME Yazılım A.Ş.\ is a mid-sized software company headquartered in the Bilkent technopark in Ankara. The company builds business-to-business SaaS products for the logistics sector, including a shipment-tracking dashboard, a route-optimisation engine and a customer-facing portal used by Turkish forwarders and a handful of European partners. Their main customers are medium-sized freight operators that need a modern web interface on top of their existing ERP systems.

The engineering organisation has around forty people split across three product squads, a platform team and a small data team. Each squad owns its services end-to-end and follows a two-week Scrum cadence with daily stand-ups, mid-sprint refinement and a Friday demo. The stack is centred on Vue.js~3 with TypeScript on the frontend, Node.js (NestJS and plain Express) on the backend, PostgreSQL as the primary database, Redis for caching and queues, and AWS (ECS Fargate, S3, CloudWatch) for hosting.

The work culture is informal and English-friendly. Code review is mandatory for every change, the trunk is protected, and deployments to staging happen on every merge through a GitHub Actions pipeline. Engineers are encouraged to spend roughly one afternoon per sprint on internal tooling, documentation or learning.

\subsubsection{Overview of the Internship}
The internship lasted five weeks, from 26 May to 27 June 2025, on the ``Dashboard'' squad. The role was primarily frontend with light backend work. Over the period I contributed to three deliverables of increasing complexity: a redesigned shipment table in the operator dashboard, a Redis-backed rate limiter for one of the public REST APIs, and a focused push to lift unit-test coverage on the billing module. In parallel I did code reviews, wrote documentation, and shadowed an on-call rotation. By the end of the internship I had eleven merged pull requests and a clear picture of how a small product team ships software in practice.

\subsection{Internship Description}

\subsubsection{Onboarding and Environment Setup}
The first week was deliberately light on coding. After HR onboarding and the security briefing I received access to GitHub, Jira, Slack and the internal wiki, and pulled the main monorepo. The dev environment is fully Dockerised: a single \texttt{docker compose up} brings up PostgreSQL, Redis, the auth service and the API gateway, after which the Vue frontend runs against them with \texttt{pnpm dev}.

The coding style is enforced by Prettier and ESLint with a custom config, plus a strict TypeScript setup with \texttt{noUncheckedIndexedAccess} turned on. Every commit goes through a pre-commit hook that runs the linter and the unit tests for the touched files. My supervisor walked me through the architecture document on day three and assigned a small ``starter'' ticket so I could practice the full PR workflow before being given real project work. Daily stand-ups are at 10:00 and run for ten minutes; refinement and demos are on Wednesday and Friday afternoons.

\subsubsection{Project 1: \textit{Logistics Dashboard UI}}
The first project was to rewrite the \texttt{ShipmentTable} component on the operator dashboard. The legacy version was a large Vue 2 Options-API component that mixed data fetching, sorting and presentation, and was difficult to extend. The goal was to split it into a thin container plus a presentational table built with the Composition API, add proper TypeScript types for the shipment domain model, and improve keyboard accessibility.

I started by reading the existing code and writing down the props and events it actually used, which turned out to be a small subset of what the file exposed. The new component takes a typed \texttt{shipments} prop and emits \texttt{select} and \texttt{sort} events; data fetching lives in a parent container that uses a Pinia store. The main lessons were practical: how to design a component API before writing the implementation, how to keep visual changes invisible to users while changing the internals, and how much accessibility work (focus order, ARIA roles, keyboard handlers) is involved in a ``simple'' table.

\subsubsection{Project 2: \textit{REST API rate limiter}}
The second project was to add a rate limiter to the public tracking API, which had started receiving aggressive automated traffic from a single integration partner. After a short investigation I proposed a sliding-window-log algorithm backed by Redis, configurable per route and per API key, returning standard \texttt{429 Too Many Requests} responses with \texttt{Retry-After} headers.

The implementation is a thin Express middleware that uses \texttt{ioredis} with a Lua script to make the window update atomic. Tests were the hardest part: unit tests with \texttt{ioredis-mock} caught logic bugs, but two subtle race conditions only appeared in integration tests against a real Redis container started with \texttt{testcontainers}. I learned that ``works on my machine'' is not a substitute for testing against the real dependency, and that middleware design benefits enormously from being a pure function of \texttt{(request, state)} so it can be exercised in isolation.

\subsubsection{Project 3: \textit{Unit test coverage push}}
The third project was less glamorous but, in hindsight, the most educational. The billing module had drifted to 62\% line coverage and the team wanted it back above 80\% before a planned refactor. My job was to write the missing tests and report any code that could not reasonably be tested.

I started with the pure functions (discount calculation, tax rounding, invoice numbering) because they are easy and have a high payoff. From there I moved into the side-effecting code, mocking the database with an in-memory repository and the time with \texttt{@sinonjs/fake-timers}. One function -- the legacy PDF generator -- was so tangled with I/O that I had to refactor it into a generator plus a writer before tests became possible; my supervisor reviewed the refactor carefully. Final coverage landed at 81\%, and I came away with a much better intuition for what makes code testable: small functions, explicit dependencies, no hidden global state.

\subsubsection{Additional Tasks and Contributions}
Outside the three projects I did roughly a dozen code reviews for teammates, mostly on the same dashboard codebase. I also rewrote the ``first-day setup'' page on the internal wiki, because several of the steps that tripped me up turned out to trip up everyone, and shadowed two on-call shifts with the platform engineer to understand how production incidents are handled (paging via PagerDuty, runbooks in the wiki, post-mortems within 48 hours).

\subsection{Conclusion}
The internship gave me a concrete answer to ``what does industrial software actually look like.'' On the hard-skills side I now have working experience with Vue~3 + TypeScript, Node.js middleware, Redis, Jest and the basics of an AWS-based deployment pipeline, and a much better grasp of testing as a design tool rather than a chore. On the soft-skills side I learned how to scope a ticket, how to ask for help efficiently in Slack, and how much value a careful code review adds to both reviewer and author.

The connection to the curriculum was direct. The Databases course made the schema discussions readable; Software Engineering made the Scrum cadence and review workflow feel familiar; Operating Systems and Networks helped me reason about the rate limiter and about debugging an HTTP issue end-to-end. If I were to do the internship again I would push earlier for harder backend tickets, take notes more systematically, and start writing tests on day one instead of after the first round of bug reports. Overall it was a useful, demanding and well-supervised five weeks, and I am grateful to ACME Yazılım and to my supervisor for the opportunity.

// Project 2 - sliding-window rate limiter (simplified excerpt).
export function rateLimit(opts: { limit: number; windowMs: number }) {
  return async function middleware(req, res, next) {
    const key = `rl:${req.apiKey}:${req.route.path}`;
    const allowed = await slidingWindow(redis, key, opts);
    if (!allowed) {
      res.set("Retry-After", String(Math.ceil(opts.windowMs / 1000)));
      return res.status(429).json({ error: "rate_limited" });
    }
    return next();
  };
}
